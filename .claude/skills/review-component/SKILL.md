# review-component

Review a React component against project standards and create a GitHub issue if problems are found.

## Instructions

You are performing a comprehensive code review of a React component. Follow these steps:

### 1. Get Component File Path

Extract the file path from $ARGUMENTS. If no path is provided, ask the user which component file to review.

### 2. Read the Component File

Use the Read tool to load the React component file.

### 3. Review Against 9-Point Checklist

Analyze the component against each of these criteria based on the project's CLAUDE.md standards:

#### 1. Component Structure
- ✅ PASS: Functional component (no class components)
- ✅ PASS: Single responsibility principle followed
- ✅ PASS: Reasonable component size (under 300 lines)
- ❌ FAIL: Class component used, multiple responsibilities, or too large

#### 2. Hook Usage
- ✅ PASS: Hooks called at top level (not in conditions/loops)
- ✅ PASS: Custom hooks used for data/business logic
- ✅ PASS: useEffect dependencies correct and not missing
- ❌ FAIL: Hooks called conditionally, logic in component instead of hooks, or missing dependencies

#### 3. State Management
- ✅ PASS: UI state only (modals, forms) in component
- ✅ PASS: Shared state via Context API (UserContext, FavoritesContext, TripContext)
- ✅ PASS: No unnecessary state variables
- ❌ FAIL: Business logic state in component, direct context imports, or redundant state

#### 4. Props and Data Flow
- ✅ PASS: Props destructured in function signature
- ✅ PASS: PropTypes or TypeScript types defined (if applicable)
- ✅ PASS: Props passed down cleanly without prop drilling
- ❌ FAIL: Props not destructured, no type checking, or excessive prop drilling

#### 5. Performance
- ✅ PASS: No unnecessary re-renders
- ✅ PASS: useMemo/useCallback used appropriately for expensive operations
- ✅ PASS: No inline function definitions in JSX (for event handlers used in lists)
- ❌ FAIL: Unnecessary re-renders, missing memoization, or inline functions causing performance issues

#### 6. Error Handling
- ✅ PASS: Loading states displayed (if component fetches data)
- ✅ PASS: Error states handled and displayed
- ✅ PASS: Try-catch blocks for async operations
- ❌ FAIL: No loading states, errors not handled, or no try-catch for async

#### 7. Naming Conventions
- ✅ PASS: PascalCase for component file (e.g., HomeScreen.jsx)
- ✅ PASS: camelCase for functions, variables, and props
- ✅ PASS: Descriptive and meaningful names
- ❌ FAIL: Incorrect casing, unclear names, or inconsistent naming

#### 8. Tailwind CSS Usage
- ✅ PASS: Only Tailwind utility classes used (no inline styles)
- ✅ PASS: Consistent spacing and sizing utilities
- ✅ PASS: Responsive classes used appropriately
- ❌ FAIL: Inline styles present, inconsistent utilities, or no responsive design

#### 9. Basic Accessibility
- ✅ PASS: Semantic HTML elements used
- ✅ PASS: Alt text on images
- ✅ PASS: ARIA labels where needed (buttons, links, forms)
- ✅ PASS: Keyboard navigation support
- ❌ FAIL: Non-semantic HTML, missing alt text, no ARIA labels, or poor keyboard navigation

### 4. Compile Results

For each checklist item:
- Note the status (✅ PASS or ❌ FAIL)
- If FAIL, provide specific line numbers and explain the issue
- Include a severity level: 🔴 Critical, 🟡 Warning, or 🔵 Info

Create a markdown table with these columns:
| Category | Status | Severity | Issue | Location |

Example:
| Category | Status | Severity | Issue | Location |
|----------|--------|----------|-------|----------|
| Hook Usage | ❌ FAIL | 🔴 Critical | useEffect missing dependency: `fetchData` | Line 45 |
| Tailwind CSS | ❌ FAIL | 🟡 Warning | Inline style used: `style={{ margin: 10 }}` | Line 78 |

### 5. Decision: Create GitHub Issue or Not

**If ANY issues are found (any ❌ FAIL results):**
- Create a GitHub issue using the GitHub MCP server
- Title: `Code Review: [filename]` (e.g., "Code Review: HomeScreen.jsx")
- Body should contain:
  - Summary: "Automated code review found [X] issues in [filename]"
  - The full markdown table of findings
  - Section for each failed category with details
  - Footer: "Generated by Claude Code review-component skill"
- Add labels: `code-review`, `needs-attention`
- If any 🔴 Critical issues exist, also add label: `priority-high`

**If NO issues are found (all ✅ PASS):**
- Skip the GitHub issue creation step
- Present a success message to the user:
  ```
  ✅ Component Review Complete: [filename]

  All checks passed! This component follows project standards.
  No GitHub issue created.
  ```

### 6. Present Results to User

Show:
1. Summary of review (X/9 checks passed)
2. The markdown table
3. If issues found: Link to created GitHub issue
4. If no issues: Success message

### Important Notes

- Be thorough but fair in assessment
- Provide specific, actionable feedback with line numbers
- Consider context - some violations may be justified
- Focus on project standards from CLAUDE.md
- Do NOT create a GitHub issue if all checks pass
- Always use the GitHub MCP server (mcp__github__create_issue) for issue creation
- Ensure GitHub MCP server is available before attempting to create an issue
